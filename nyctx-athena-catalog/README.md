# nyctx-athena-catalog

Athena DDL and cost-aware SQL examples for the NYC Taxi Silver lakehouse layer.

This folder is intentionally small. It focuses on the production-like Silver
Parquet table generated by the Airflow and Glue pipeline.

## S3 Layout

The current Silver layout is Hive-style partitioned by year and month:

```text
s3://nyc-taxi-lakehouse-tntk/silver/yellow_taxi/year=YYYY/month=MM/
```

Example processed partitions:

```text
2019-01
2020-01
2020-03
2020-04
2020-05
2021-01
2022-01
2023-01
2024-01
```

## Create Catalog Objects

Run these in Athena, using the project workgroup:

```text
ddl/create_database.sql
ddl/create_silver_yellow_taxi.sql
```

The Silver table uses partition projection, so you do not need to run manual
`ALTER TABLE ADD PARTITION` for every processed month.

Partition columns are strings:

```text
year  STRING
month STRING
```

This matches the S3 folder names such as `year=2024/month=01/` and makes query
filters explicit.

## Safe Smoke Test

Start with:

```text
queries/00_smoke_test.sql
```

It reads one partition only:

```sql
WHERE year = '2024'
  AND month = '01'
```

Then check Athena Query history and confirm the `Data scanned` value is small.

## Airflow Integration

The Airflow DAG runs only the catalog setup and validation control queries:

```text
setup_athena_catalog
validate_silver_athena
```

`setup_athena_catalog` executes:

```text
ddl/create_database.sql
ddl/create_silver_yellow_taxi.sql
```

`validate_silver_athena` reads `config/recovery_sample_months.txt` and runs one
partition-filtered row-count query per month:

```sql
SELECT COUNT(*) AS trip_count
FROM nyc_taxi_lakehouse.silver_yellow_taxi
WHERE year = '<YYYY>'
  AND month = '<MM>';
```

Each validation query fails the task if the month returns zero rows.

## Cost Rules

Athena charges by data scanned. Keep queries safe by following these rules:

- Do not use `SELECT *`.
- Always filter `year` and `month`.
- Keep trend queries bounded to a short list of known partitions.
- Prefer aggregations over raw row inspection.
- Use `ORDER BY` only on aggregated outputs or with `LIMIT`.

See [docs/query_cost_guardrails.md](docs/query_cost_guardrails.md).

## Query Checklist

| Query | Scan guardrail |
|---|---|
| `00_smoke_test.sql` | Single partition, explicit columns, `LIMIT` |
| `01_monthly_trip_summary.sql` | Bounded recovery month list only |
| `02_payment_type_summary.sql` | Single partition aggregation |
| `03_hourly_demand_pattern.sql` | Single partition aggregation |
| `04_top_pickup_zones.sql` | Single partition aggregation, `LIMIT 25` |

The query examples are intentionally not part of the Airflow DAG. They are for
manual validation, portfolio screenshots, BI exploration, or future dbt Gold
model design.

## Removed Files

The previous catalog folder contained single-month Bronze/reference/profiling
and validation SQL from the initial demo phase. Those files were removed because:

- they targeted only `2024-01`;
- some used broad or unsafe scans;
- some used `SELECT *`;
- manual partition SQL does not scale as more months are processed;
- the current portfolio layer should focus on the production Silver Parquet
  table and cost-aware Athena query patterns.
